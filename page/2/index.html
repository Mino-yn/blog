背景

方案

  • 备份恢复：先用全量备份恢复实例，再利用增量 binlog 备份恢复到误操作之前的状态，费时费力！
  • 第三方工具：binlog2sql（支持 DML Delete/Update 等语句回滚，不支持 DDL Drop 等语句回滚）

binlog2sql 工具实战

支持

  • MySQL 5.6/5.7
  • Delete、Update

要求

  • MySQL 配置要求
  • MySQL 权限要求**：SELECT、REPLICATION SLAVE、REPLICATION CLIENT
  • 申请特权账号，登录服务器

操作步骤

  • 安装 pip
  • 安装 binlog2sql
  • 根据误操作时间，确定 binlog 文件
  • 定位误执行 SQL
  • 解析出回滚 SQL，并检查回滚 SQL 是否正确

Tips：回滚条数可以使用 %s/pattern/&/g 进行统计，pattern 是要搜索的关键字。

  • 确认回滚 SQL 正确，执行回滚语句

Xtrabackup · Error

发表于 2020-06-17

Error 124

背景

使用 innobackupex 全备 mysql 数据库产生 returned OS error 124 错误

错误详情

解决方法

  • 该错误是指 mysql 进程打开的文件数超出可打开最大文件数量的限制，查看 mysql 数据文件目录下的文件总数

查看 innodb_open_files 参数

innodb_open_files 限制 innodb 能打开的表的数量，这里 innodb_open_files 的默认值为 2000，而实际文件数量为3833，已超出限制。 该参数无法动态修改，需修改 my.cnf 配置文件。

在[mysqld]下加入innodb_open_files=10240

  • 重新进行备份，但仍报相同错误。怀疑是系统可打开最大文件数目的问题

open files 显示为1024, 重新设置 open files 的值

重新备份，问题解决

删除大文件，磁盘空间未释放

发表于 2020-05-18

背景

HDY 数据库所在服务器到期，需要腾出磁盘空间，进行数据库备份迁移，查看文件大小，发现有一个非常大的err日志文件 326GB, 为腾空间，决定将其删除。

过程

删除后查看磁盘空间，发现空间未释放。应该有程序还在操作这个文件，指针未释放

删除进程

总结

  • 方法一

    在删除之前 先清空文件，再删除文件 这样能快速释放磁盘空间。

  • 方法二

    若文件已删除，但磁盘空间未被释放，很可能是此时这个文件被别的进程所占用，kill 掉占用文件的进程号即可解决。注意：kill 进程之前，确认一下删除的是什么进程，防止删除了重要的进程，造成不必要的麻烦
