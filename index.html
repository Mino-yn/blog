MySQL · ibtmp1文件过大

发表于 2020-07-23

背景

项目经理反应服务器上有个500多G的文件，占用大量磁盘空间

分析

ibtmp1 文件是非压缩的 innodb 临时表的独立表空间。通过 innodb_temp_data_file_path 参数指定文件的路径。文件名和大小，默认配置为 ibtmp1:12M:autoextend ，也就是说在支持大文件的系统这个文件大小是可以无限增长的。

查看参数 innodb_temp_data_file_path

果然没有限制大小

解决方法

释放ibtmp1 并限制文件大小

1、修改配置文件

2、关闭数据库实例

​ 关闭后 ibtmp1 文件会自动清理

3、启动数据库是理科

3、启动后查询参数是否生效

优化策略

调大 tmp_table_size 、max_heap_table_size 参数

建议 : 256M

总结

为了避免 ibtmp1 文件无止境的暴涨占用过多的磁盘空间,可以修改 innodb_temp_data_file_path 参数，限制其文件最大尺寸。

ibtmp1 加上限制最大值。innodb_temp_data_file_path = ibtmp1:12M:autoextend:max:5G，mysql 会反复利用。

一、MySQL 在什么情况下会创建临时表(Internal Temporary Table Use in MySQL)？

  • GROUP BY 无索引字段或GROUP BY+ ORDER BY 的子句字段不一样时

  • order by 与distinct 共用，其中distinct与order by里的字段不一致（主键字段除外）

  • UNION查询（MySQL5.7后union all已不使用临时表）

  • insert into select …from …

  • 数据表中包含blob/text列

具体参考：https://dev.mysql.com/doc/refman/5.7/en/internal-temporary-tables.html

二、判断sql是否使用了临时表

EXPLAIN 查看执行计划结果的 Extra 列中，如果包含 Using Temporary 就表示会用到临时表。

三、临时表有关的参数有哪些？

四、 ibtmp1 为何变大

​ 有大量慢sql，而且有很多语句需要排序。

​ 如果临时表中需要存储的数据量超过了上限（ tmp-table-size 或 max-heap-table-size 中取其大者），这时候就需要生成基于磁盘的临时表了。也就是放在 innodb_temp_data_file_path指定的临时表空间中。

五、如何优化

​ 内部临时表 (Internal Temporary Table) 用于排序，分组，当需要的存储空间超过 tmp-table-size 上限的时候，使用临时表空间。临时表空间是磁盘，速度比不上内存，所以可通过加大 tmp_table_size 和 ``max-heap-table-size来优化需要使用临时表的SQL语句。 建议256M`

  • 第三方工具：binlog2sql（支持 DML Delete/Update 等语句回滚，不支持 DDL Drop 等语句回滚）

  • MySQL 配置要求

  • 申请特权账号，登录服务器

操作步骤

  • 安装 pip
  • 安装 binlog2sql
  • 根据误操作时间，确定 binlog 文件

  • 解析出回滚 SQL，并检查回滚 SQL 是否正确

Tips：回滚条数可以使用 %s/pattern/&/g 进行统计，pattern 是要搜索的关键字。
