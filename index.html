MySQL主要文件类型

配置文件

  • 当MySQL实例启动时，MySQL会先去读一个配置参数文件，用来寻找数据库的各种文件所在位置以及指定某些初始化参数，这些参数通常定义了某种内存结构有多大等设置。默认情况下，MySQL实例会按照一定的次序去取，你只需通过命令mysql –help|grep my.cnf来寻找即可。

    MySQL参数文件的作用和Oracle的参数文件极其类似；不同的是，Oracle实例启动时若找不到参数文件，是不能进行装载（mount）操作的。MySQL稍微有所不同，MySQL实例可以不需要参数文件，这时所有的参数值取决于编译MySQL时指定的默认值和源代码中指定参数的默认值。但是，如果MySQL在默认的数据库目录下找不到mysql架构，则启动同样会失败，你可能在错误日志文件中找到如下内容：

  • MySQL的很多性能优化工作、主从复制等都是在调整参数、主从复制等都是在调整参数以适应不同时期不同数量级的数据，因此理解my.cnf中的参数非常重要，只有理解了参数设置，才可能对数据库进行某些方面的调优工作。

  • Mysql参数文件中的参数可以分为2种类型：动态（dynamic）参数和静态参数（staitic）
    动态参数意味着可以在mysql实例运行中进行修改，修改后无需重启数据库即可生效
    生效范围分为：global和session。

    静态的说明在整个mysql实例运行期间不得进行修改，就类似一个只读的read only，需要修改数据库配置文件，然后重启数据库实例才可生效

日志文件

日志文件记录了影响 mysql 数据库的各种类型活动，常见的日志文件有错误日志、二进制日志、慢查询日志、全查询日志、redo日志、undo日志

错误日志

错误日志对mysql的启动、运行、关闭过程进行了记录，

在遇到问题时候，第一时间应该查看这个错误日志文件，

错误日志也不仅仅记录出错信息，还记录了一些警告信息以及正确信息，总的来说，可以在错误日志文件里面看到一些数据库启动信息，以及告警信息还有就是报错信息

二进制文件

记录对MySQL数据库真正执行更改的所有操作不包含没有修改任何数据的语句：select、show

如果你还想记录select和show的话，你就需要查看全查询日志，另外binlog还包括了执行数据库更改操作时间和执行时间等信息。

二进制的主要作用有如下2个：
1.主从复制，在主服务器中把所有修改数据的操作记录到binlog中，通过网路发送给从服务器，从服务器将binlog中记录的操作重新执行一遍，从而实现主从同步。

2、恢复 ：某些数据的恢复需要二进制日志，在全库文件恢复后，可以在此基础上通过二进制日志进行基于时间点的恢复。还可以进行通过binlog日志实现对误操作的恢复工作。

日志格式：

Statement：每一条会修改数据的sql都会记录在binlog中。

Row:不记录sql语句上下文相关信息，仅保存哪条记录被修改。
如：更新语句更新10行记录时，所有10行更新信息都会被写入日志，

基于Statement，只有更新语句的SQL写入日志。默认格式ROW

Mixedlevel: 是以上两种level的混合使用，一般的语句修改使用statment格式保存binlog，如一些函数，statement无法完成主从复制的操作，则采用row格式保存binlog。

存储数据

我们也可以自己指定使用系统表空间还是独立表空间来存储数据，这个功能由启动参数innodb_file_per_table控制。

当innodb_file_per_table的值为0时，代表使用系统表空间，表中的数据都会被存储到这个 系统表空间。；当innodb_file_per_table的值为1时，代表使用独立表空间，每一个表建立一个独立表空间，就会生成一个ibd文件，也就是说我们创建了多少个表，就有多少个独立表空间。

MySQL · ibtmp1文件过大

发表于 2020-07-23

背景

项目经理反应服务器上有个500多G的文件，占用大量磁盘空间

分析

ibtmp1 文件是非压缩的 innodb 临时表的独立表空间。通过 innodb_temp_data_file_path 参数指定文件的路径。文件名和大小，默认配置为 ibtmp1:12M:autoextend ，也就是说在支持大文件的系统这个文件大小是可以无限增长的。

查看参数 innodb_temp_data_file_path

果然没有限制大小

解决方法

释放ibtmp1 并限制文件大小

1、修改配置文件

2、关闭数据库实例

​ 关闭后 ibtmp1 文件会自动清理

3、启动数据库是理科

3、启动后查询参数是否生效

优化策略

调大 tmp_table_size 、max_heap_table_size 参数

建议 : 256M

总结

为了避免 ibtmp1 文件无止境的暴涨占用过多的磁盘空间,可以修改 innodb_temp_data_file_path 参数，限制其文件最大尺寸。

ibtmp1 加上限制最大值。innodb_temp_data_file_path = ibtmp1:12M:autoextend:max:5G，mysql 会反复利用。

一、MySQL 在什么情况下会创建临时表(Internal Temporary Table Use in MySQL)？

  • GROUP BY 无索引字段或GROUP BY+ ORDER BY 的子句字段不一样时

  • order by 与distinct 共用，其中distinct与order by里的字段不一致（主键字段除外）

  • UNION查询（MySQL5.7后union all已不使用临时表）

  • insert into select …from …

  • 数据表中包含blob/text列

具体参考：https://dev.mysql.com/doc/refman/5.7/en/internal-temporary-tables.html

二、判断sql是否使用了临时表

EXPLAIN 查看执行计划结果的 Extra 列中，如果包含 Using Temporary 就表示会用到临时表。

三、临时表有关的参数有哪些？

四、 ibtmp1 为何变大

​ 有大量慢sql，而且有很多语句需要排序。

​ 如果临时表中需要存储的数据量超过了上限（ tmp-table-size 或 max-heap-table-size 中取其大者），这时候就需要生成基于磁盘的临时表了。也就是放在 innodb_temp_data_file_path指定的临时表空间中。

五、如何优化

​ 内部临时表 (Internal Temporary Table) 用于排序，分组，当需要的存储空间超过 tmp-table-size 上限的时候，使用临时表空间。临时表空间是磁盘，速度比不上内存，所以可通过加大 tmp_table_size 和 ``max-heap-table-size来优化需要使用临时表的SQL语句。 建议256M`

MySQL · 误操作场景下数据快速恢复

发表于 2020-07-09

背景

线上核心任务，delete/update 操作错误，如何快速恢复？

方案

  • 备份恢复：先用全量备份恢复实例，再利用增量 binlog 备份恢复到误操作之前的状态，费时费力！
  • 第三方工具：binlog2sql（支持 DML Delete/Update 等语句回滚，不支持 DDL Drop 等语句回滚）

binlog2sql 工具实战

支持

  • MySQL 5.6/5.7
  • Delete、Update

要求

  • MySQL 配置要求
  • MySQL 权限要求**：SELECT、REPLICATION SLAVE、REPLICATION CLIENT
  • 申请特权账号，登录服务器

操作步骤

  • 安装 pip
  • 安装 binlog2sql
  • 根据误操作时间，确定 binlog 文件
  • 定位误执行 SQL
  • 解析出回滚 SQL，并检查回滚 SQL 是否正确

Tips：回滚条数可以使用 %s/pattern/&/g 进行统计，pattern 是要搜索的关键字。

  • 确认回滚 SQL 正确，执行回滚语句

工具

注意事项

  • 还是先恢复再去面壁思过吧！
  • 手工执行 delete、update 语句前，先执行 selete 语句。
  • drop 或者 truncate 语句造成的误操作，通过备份来恢复。
